# Gada Terminal Roadmap

**Goal:** Build a cross-platform Electron desktop app that wraps the Claude Code CLI with a real-time monitoring sidebar for tool calls, cost, context usage, and permission approval.
**Approach:** Windows-first incremental delivery, cutting vertically through all layers each slice. Start with a working terminal, add the launch screen, then layer in the HTTP hook server and each sidebar panel one at a time before finishing with persistence polish and the plugin framework.
**Created:** 2026-04-06

## Slices

- [x] Slice 1 — Working Terminal: A Windows desktop app opens with a full-fidelity terminal running Claude via xterm.js + node-pty. You can type, interact with Claude, copy/paste, drag files in, and the terminal resizes with the window.
- [x] Slice 2 — Launch Screen: Before Claude starts, a GUI card lets you pick a working directory (with recent folders), choose model/effort/permission mode/extra args, and click "Start Claude". Settings persist per folder between sessions.
- [x] Slice 3 — Live Tool Feed: A sidebar appears with a live feed of every tool Claude invokes — file reads, bash commands, web searches — each shown with a timestamp, target, and running/done/failed status badge. Requires the internal HTTP hook server.
- [x] Slice 4 — Cost & Context Panels: Two new sidebar sections show real-time token counts, cost per request, and a color-coded context window progress bar — all updated live as Claude works via OTLP telemetry.
- [x] Slice 5 — Permission Approval: When Claude wants to run a tool, a card appears in the sidebar. Claude waits while you choose Allow, Allow for session, or Deny. Auto-approved reads are transparent; risky actions require a click.
- [x] Slice 6 — Persistence & Polish: Accent color, panel reorder/hide, recent plugins all survive restarts. The app feels complete and personalized. → [plan](../slices/2026-04-06-slice-6-persistence-polish.md)
- [x] Slice 7 — Plugin Loader: A `plugins/` directory is scanned on startup. Each plugin's `plugin.json` manifest is parsed and its entry point loaded into a sandboxed renderer context with a read-only PanelAPI (event subscriptions only). A sample plugin renders a panel. Drop a folder in, see a new panel appear. → [plan](../slices/2026-04-07-slice-07-plugin-loader.md)
- [x] Slice 8 — Full PanelAPI & Permissions: The PanelAPI gains write capabilities — terminal input, Claude messages, process spawning, external HTTP. Plugins declare required permissions in their manifest. First use of a capability triggers a user approval prompt (same pattern as tool approval); grants persist. A sample plugin writes to the terminal after approval. → [plan](../slices/2026-04-10-slice-08-full-panel-api-permissions.md)
- [x] Slice 9 — Plugin Management UI: A settings screen lists installed plugins with name, version, enabled/disabled toggle, and individually revocable permissions. Disabling a plugin removes its panel immediately. → [plan](../slices/2026-04-10-slice-09-plugin-management-ui.md)
- [-] Slice 10 — Linux Support (deferred — development resources): The app runs correctly on Linux with titleBarOverlay accent color, correct Claude path resolution, and DEB/RPM packaging. → [plan](../slices/2026-04-07-slice-07-linux-support.md)
- [-] Slice 11 — macOS Support (deferred — development resources): The app runs correctly on macOS with hidden-inset titlebar, custom HTML accent bar styled with the accent color, Homebrew Claude path resolution, Cmd+V paste, and DMG packaging.
- [x] Slice 12 — Sidebar Redesign & Widget Management: A right-hand sidebar is added. Widgets can be dragged between left and right sides. A sidebar automatically hides when it contains no widgets. The current show/hide buttons are replaced by a settings popup (always accessible even when both sidebars are empty) that lists all widgets with visibility toggles. Depends on: Slice 6. → [plan](../slices/2026-04-10-slice-12-sidebar-redesign.md)
- [x] Slice 13 — Permission Manual Override: The permission widget gains an override button. Pressing it releases the hook — equivalent to a timeout — causing Claude Code to fall back to displaying the full interactive permission question directly in the terminal. The user answers there as normal. Depends on: Slice 5. → [plan](../slices/2026-04-10-slice-13-permission-manual-override.md)
- [ ] Slice 14 — Error Log Widget: A widget that subscribes to `hook:tool-event` and surfaces only `PostToolUseFailure` events as a timestamped list showing the tool name and error message. Clears on session start. No capabilities required.
- [ ] Slice 15 — WidgetAPI Context: `WidgetAPI` gains a `getContext()` method returning the active working directory path. This is a prerequisite for any widget that needs to scope data or commands to the current project. No capabilities required.
- [ ] Slice 16 — Project Notes Widget: A plain textarea widget for jotting notes about the current project. Content persists in `localStorage` keyed by the working directory from `WidgetAPI.getContext()` so each project has its own independent notes that survive restarts. No capabilities required. Depends on: Slice 15.
- [ ] Slice 17 — Git Status Widget: A widget that runs `git status --short` and `git log --oneline -5` in the active working directory (from `WidgetAPI.getContext()`) via `process:spawn` and renders the results. Refreshes on load and via a manual refresh button. Requires `process:spawn` capability. Depends on: Slice 15.
- [ ] Slice 18 — Quick Prompts Widget: A widget showing a user-editable list of saved prompt snippets. Each entry has a label and message body, stored in `localStorage`. Clicking a prompt sends it to Claude via `claude:message`. The UI allows adding, editing, and removing entries inline. Requires `claude:message` capability.
- [ ] Slice 19 — Pomodoro Timer Widget: A focus timer widget with configurable work and break durations (stored in `localStorage`). Shows a countdown and cycles automatically between work and break intervals. Tracks completed pomodoros and total focused time for the session. No capabilities required.
- [ ] Slice 20 — Widget OS Targeting: The widget manifest gains an optional `os` field accepting an array of platform strings (`win32`, `darwin`, `linux`) matching Node's `process.platform`. Widgets that declare `os` are silently excluded on non-matching platforms. Widgets with no `os` field are treated as cross-platform. The widget management UI indicates platform-restricted widgets if shown on a matching OS.
- [ ] Slice 21 — Open Terminal Widget: A widget with a button that opens a terminal in the current working directory as a detached, independent process. On load it probes for available terminals (e.g. Windows Terminal, PowerShell, Command Prompt) and a settings popover lets the user choose between those found; the selection is stored in `localStorage`. Declared as `"os": ["win32"]`. Requires a new `shell:launch` capability in the main process that spawns the process with `detached: true` and unrefs it, rather than waiting for exit like `process:spawn` does. Depends on: Slice 15, Slice 20.
